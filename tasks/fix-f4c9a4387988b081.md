# fix-f4c9a4387988b081 — Double-counting when old and new time slots overlap during reschedule

**Weight:** 0.0000 (share of project budget)
**Reward:** 0 TBLBOT

If a user reschedules to a time slot that overlaps with their original slot (e.g., shifting from 18:00–19:30 to 18:30–20:00), `allocateFirstFit` will count the original booking's tables (still allocated, status "rescheduled") AND attempt to allocate new tables for the same party. This causes a double-count of capacity — the same physical tables appear consumed twice. The new booking may fail with "Not enough available seats" even though enough seats exist, because the old booking's tables are still blocking.

This is a consequence of Bug 2 (zombie allocations) but manifests as a distinct user-facing error.
