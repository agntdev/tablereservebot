# fix-a215698097c3c321 — Original booking's allocated tables are never freed — zombie allocations cause permanent capacity loss

**Weight:** 0.0000 (share of project budget)
**Reward:** 0 TBLBOT

When `/reschedule` marks the original booking as "rescheduled" (`bot.ts:663`), it calls `storage.updateBookingStatus` which only sets the `status` field — it does NOT clear `allocated_tables`. The `allocateFirstFit` function (`allocate.ts:57`) includes "rescheduled" in its `activeStatuses` set, so the original booking's tables continue to block availability checks even after the reschedule is completed.

After a successful reschedule, a NEW booking is created with NEW table allocations (`bot.ts:271–277`), but the OLD booking's tables are never released. They remain permanently allocated to a "rescheduled" booking that nobody owns. Over time, every `/reschedule` leaks table capacity.

**Fix**: Either (a) clear `allocated_tables` (set to `[]`) when marking the original booking as "rescheduled", or (b) delete the original booking entirely after a successful reschedule, or (c) remove "rescheduled" from `activeStatuses` in `allocate.ts` and treat rescheduled bookings as freed.
